Future Improvements

Purpose
- This file is meant to help future work sessions quickly understand what the admin
  panel is already doing and what should be improved next.
- The project is currently a custom mini-CMMS built on top of Jira tickets.
- Jira tickets are the operational source, while the admin panel adds machine-level
  metadata, repair/manual cost tracking, maintenance planning, and statistics.

Current Admin Responsibilities
- Costs:
  filter Jira tickets, attach repair costs to tickets, and manage manual machine costs
- Inventory:
  manage equipment metadata per machine key
- Asset detail:
  show one-machine rollup across tickets, repair/manual costs, maintenance, and metadata
- Maintenance:
  plan, edit, complete, delete, and repair maintenance items in a calendar UI
  while syncing linked maintenance items with Jira
- Statistics:
  derive rollups from Jira tickets plus stored admin data
- Users:
  manage app users, limited to super-admin access
  self-service password change for other admins through My Account

Important Data Model Notes
- Machine identity is centered around machineKey, usually in category::subcategory form.
- Jira issues are mapped to machines through parsed issue asset metadata.
- Asset/equipment details are stored separately from Jira and are merged in admin views.
- Planned maintenance currently uses item-based scheduling, not reusable PM templates.
- Planned maintenance rows can now store Jira linkage:
  jiraIssueId, jiraIssueKey, jiraIssueUrl
- Planned maintenance rows now preserve the selected due timestamp, not only a date.
  Calendar grouping and due-soon logic still work off the local day portion.
- Maintenance creator handling is now more consistent:
  new items use the signed-in session user
  legacy items without createdBy can be backfilled when edited
- Planned maintenance create/edit/delete is no longer local-only.
  New items are created in Jira first and then stored locally.
- Current Jira maintenance sync defaults:
  project key: MECH
  issue type: Submit a request or incident
  target status after create: To Do List
  labels: planned-maintenance, admin-calendar
- Ticket sync is stale-while-revalidate and supports manual full refresh.
- If a linked Jira maintenance issue is deleted outside the app, the next maintenance
  load repairs local state by removing the stale local row.
- Linked maintenance delete from the admin panel now deletes the Jira issue first and
  then removes the local row.
- Mutating admin APIs now enforce admin auth plus same-origin checks.
- Bootstrap admin login should be treated as setup-only and is now gated by
  ALLOW_ADMIN_BOOTSTRAP=true and zero existing users.
- User-management boundary now works like this:
  ignven is the super-admin identity
  only super-admin can create, edit, delete, or fully list other users
  other admins can open My Account and change only their own password

Maintenance Sync Status
- Implemented:
  create in Jira + local DB
  edit in local UI + Jira
  complete/reopen in local UI + Jira field update/comment
  delete in local UI + Jira delete
  repair local stale rows when Jira issue no longer exists
  selected-recipient email notifications through Resend
  localized EN/LT maintenance emails
  reply-to support for Resend notifications
  creator included in maintenance emails
  due date and time included in maintenance emails
- Not implemented:
  Jira-side edits changing local maintenance rows automatically
  Jira-side status transitions mapping back into richer local workflow states
  orphan-state UI; current repair path removes stale linked rows instead

Priority Order
- Highest priority improvements should deepen maintenance operations, not just add
  more charts or visual polish.
- If choosing between analytics and workflow depth, prefer workflow depth first.

What now exists
- dedicated route: app/admin/assets/[machineKey]/page.tsx
- shared loader hook: app/admin/hooks/useAdminAssetDetail.ts
- asset-level view of:
  Jira breakdown tickets
  repair costs tied to tickets
  manual costs
  planned maintenance
  completed maintenance
  asset metadata
- entry points from existing admin views:
  costs
  inventory
  statistics
Follow-up improvements for this page
- localize the newly added asset-detail strings properly in Lithuanian
- consider adding direct edit actions from the asset page instead of only read-first views
- consider linking from maintenance lists/calendar directly into the same asset detail page
- if users still need audit-style chronology, consider a dedicated asset activity log
  instead of reintroducing a duplicate "recent activity" panel beside the ticket list
- decide whether the asset page should eventually own reusable asset-level actions
  like edit metadata, add manual cost, or create maintenance directly in place

2. Maintenance workflow states
Why
- This was partially completed already.
What now exists
- planned
- in progress
- waiting for parts
- completed
- cancelled
What still needs attention
- Review whether the current modal/footer should expose more status transitions again
  in a better UX, or whether status changes should stay intentionally minimal.
- Decide how each local status should map to Jira issue status, comment, label, or
  custom field behavior long term.
- Review whether statistics and asset views should expose richer status breakdowns.

3. Preventive maintenance templates
Why
- Current planning is item-based. Reusable templates are needed for real PM programs.
What to add
- reusable PM templates
- recurrence rules: weekly, monthly, custom
- checklist steps
- estimated labor
- expected cost
Likely implementation direction
- Add new DB model(s) for templates and generated maintenance items.
- Keep generated items separate from templates.
- Avoid baking recurrence logic directly into page components.

4. Asset master data expansion
Why
- Current inventory data is useful but too thin for long-term asset management.
What to add
- internal asset ID
- location
- install date
- warranty end date
- vendor/service provider
- criticality
- notes/documents
Likely implementation direction
- Extend asset/equipment schema and admin equipment APIs.
- Keep machineKey as the main join key unless a new stronger asset ID is introduced.

5. Parts and spare inventory
Why
- This is one of the largest functional gaps compared with common CMMS products.
What to add
- part number
- stock count
- minimum stock
- supplier
- asset compatibility
- parts used on maintenance/repair jobs
Suggested scope
- Start simple. First track parts and usage history before trying to build full
  procurement or purchasing flows.

6. Notifications and reminders
Why
- Users need operational prompts, not only dashboards.
What to add
- due soon / overdue maintenance alerts
- reminders for incomplete asset details
- reminders for missing inventory information
What now exists
- planned maintenance create/update/reminder email notifications through Resend
- due-soon reminder cron route:
  app/api/cron/planned-maintenance-due-soon/route.ts
What still needs attention
- decide whether asset-detail / inventory reminder notifications should be in-app,
  email, or both
- improve admin-visible error reporting if external email delivery fails
- consider adding delivery/audit information if operators need traceability

10. Manual full refresh UX
Why
- Users can force a full Jira resync now, but the UI does not explain sync behavior.
What to add
- last sync timestamp
- indication of cached vs refreshed state
- visible sync progress/status when a full refresh runs

11. Jira maintenance sync polish
Why
- Maintenance now writes through to Jira, but the integration is still intentionally
  simple and should be hardened as a product feature.
What to do
- Store or expose clearer sync-state UI when Jira create/update/delete fails
- Consider showing the Jira issue key directly in maintenance cards/modals
- Decide whether repair should hard-delete stale local rows forever or move them into
  a softer orphaned/archived state
- Review whether more admin write routes should share a single centralized same-origin
  helper/middleware pattern instead of route-by-route checks
- Decide whether raw maintenance delete in Jira should remain allowed or whether the
  app should prefer closing/cancelling work instead
