Future Improvements

Purpose
- This file is meant to help future work sessions quickly understand what the admin
  panel is already doing and what should be improved next.
- The project is currently a custom mini-CMMS built on top of Jira tickets.
- Jira tickets are the operational source, while the admin panel adds machine-level
  metadata, repair/manual cost tracking, maintenance planning, and statistics.

Current Admin Structure
- Main page: app/admin/page.tsx
- Shared admin utilities/types: app/admin/adminShared.ts
- Presentational sections:
  app/admin/components/CostsSection.tsx
  app/admin/components/InventorySection.tsx
  app/admin/components/MaintenanceSection.tsx
  app/admin/components/StatisticsSection.tsx
  app/admin/components/AdminSidebar.tsx
  app/admin/components/AdminFilters.tsx
- Admin hooks:
  app/admin/hooks/useAdminSession.ts
  app/admin/hooks/useAdminFilters.ts
  app/admin/hooks/useAdminAssetData.ts
  app/admin/hooks/useAdminMaintenance.ts

Current Admin Responsibilities
- Costs:
  filter Jira tickets, attach repair costs to tickets, and manage manual machine costs
- Inventory:
  manage equipment metadata per machine key
- Maintenance:
  plan, edit, complete, and delete maintenance items in a calendar UI
- Statistics:
  derive rollups from Jira tickets plus stored admin data
- Users:
  manage app users, limited to super-admin access

Important Data Model Notes
- Machine identity is centered around machineKey, usually in category::subcategory form.
- Jira issues are mapped to machines through parsed issue asset metadata.
- Asset/equipment details are stored separately from Jira and are merged in admin views.
- Planned maintenance currently uses item-based scheduling, not reusable PM templates.
- Ticket sync is stale-while-revalidate and supports manual full refresh.

Priority Order
- Highest priority improvements should deepen maintenance operations, not just add
  more charts or visual polish.
- If choosing between analytics and workflow depth, prefer workflow depth first.

1. Asset detail page
Why
- This would make each machine understandable in one place, which is one of the most
  valuable patterns in mature CMMS tools.
What it should show
- Jira breakdown tickets
- repair costs tied to tickets
- manual costs
- planned maintenance
- completed maintenance
- asset metadata
- activity/change history
Likely implementation direction
- Add a dedicated admin asset route, probably based on machineKey.
- Reuse data from useAdminAssetData and useAdminMaintenance where possible.

2. Maintenance workflow states
Why
- Current maintenance handling is too binary and does not reflect real maintenance flow.
What to add
- planned
- in progress
- waiting for parts
- completed
- cancelled
Likely implementation direction
- Extend planned maintenance storage/API to persist a richer status field.
- Update the calendar, modal, and statistics logic to use statuses instead of only
  active/completed state.

3. Preventive maintenance templates
Why
- Current planning is item-based. Reusable templates are needed for real PM programs.
What to add
- reusable PM templates
- recurrence rules: weekly, monthly, custom
- checklist steps
- estimated labor
- expected cost
Likely implementation direction
- Add new DB model(s) for templates and generated maintenance items.
- Keep generated items separate from templates.
- Avoid baking recurrence logic directly into page components.

4. Asset master data expansion
Why
- Current inventory data is useful but too thin for long-term asset management.
What to add
- internal asset ID
- location
- install date
- warranty end date
- vendor/service provider
- criticality
- notes/documents
Likely implementation direction
- Extend asset/equipment schema and admin equipment APIs.
- Keep machineKey as the main join key unless a new stronger asset ID is introduced.

5. Parts and spare inventory
Why
- This is one of the largest functional gaps compared with common CMMS products.
What to add
- part number
- stock count
- minimum stock
- supplier
- asset compatibility
- parts used on maintenance/repair jobs
Suggested scope
- Start simple. First track parts and usage history before trying to build full
  procurement or purchasing flows.

6. Notifications and reminders
Why
- Users need operational prompts, not only dashboards.
What to add
- due soon / overdue maintenance alerts
- reminders for incomplete asset details
- reminders for missing inventory information
Suggested scope
- Start in-app first before email or external notification channels.

7. Statistics improvements
Why
- Statistics are useful, but not the highest-priority gap compared with workflow depth.
What to do
- Move remaining statistics derivations into a dedicated hook, likely useAdminStatistics.
- Consider richer asset-level cost and downtime reporting only after workflow depth
  improves.

8. Activity log localization
Why
- Some activity labels are still hardcoded in English.
What to do
- Replace strings like Activity, Created, Updated, and Completed with translated keys.
- Review the admin panel for any remaining fallback-key or hardcoded-text leaks.

9. Admin page cleanup
Why
- app/admin/page.tsx is much smaller than before, but still owns some cross-section
  derived state.
What to do
- Keep page.tsx mainly as a composition layer.
- Move further section-specific derivations into hooks if they start growing again.

10. Manual full refresh UX
Why
- Users can force a full Jira resync now, but the UI does not explain sync behavior.
What to add
- last sync timestamp
- indication of cached vs refreshed state
- visible sync progress/status when a full refresh runs

How Future Sessions Should Use This File
- Read this file first when asked what to improve next in the admin panel.
- Treat items 1 through 6 as more valuable than adding extra statistics or visuals.
- Before implementing a larger item, inspect the related admin hook/component files
  listed above instead of rebuilding logic from scratch.
